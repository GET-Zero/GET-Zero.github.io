GET-Zero

Graph Embodiment Transformer for
Zero-shot Embodiment Generalization

GET-Zero enables zero-shot control of previously unseen robot designs using a single set of network weights.

Abstract

This paper introduces GET-Zero, a model architecture and training procedure for learning an embodiment-aware control policy that can immediately adapt to new hardware changes without retraining. To do so, we present Graph Embodiment Transformer (GET), a transformer model that leverages the embodiment graph connectivity as a learned structural bias in the attention mechanism. We use behavior cloning to distill demonstration data from embodiment-specific expert policies into an embodiment-aware GET model that conditions on the hardware configuration of the robot to make control decisions. We conduct a case study on a dexterous in-hand object rotation task using different configurations of a four-fingered robot hand with joints removed and with link length extensions. Using the GET model along with a self-modeling loss enables GET-Zero to zero-shot generalize to unseen variation in graph structure and link length, yielding a 20% improvement over baseline methods.

Graph Embodiment Transformer (GET)

We introduce Graph Embodiment Transformer (GET), and embodiment-aware policy architecture based on the transformer encoder that makes control decisions according to the robot hardware configuration. We start with an embodiment graph (left) whichs consists of joints as nodes and links as directed edges. Next, we construct separate transformer tokens for each joint, which contain local state and hardware information. These joint tokens then communicate through a modified attention mechanism in the transformer encoder. In particular, we use both undirected (Spatial Bias) and directed (Parent-Child Bias) graph features as a structural bias in the attention scores of the self-attention layers. The main idea is to modulate the communication between joints according to how they are connected in the robot. Lastly, the model outputs per-joint actions and forward kinematics pose estimates for each joint (self-modeling). We observe that self-modeling is a useful meta task that improves zero-shot embodiment generalization.

Results

We apply the embodiment-aware GET-Zero policy to an in-hand object rotation task where a dexterous hand continuously rotates a cube. In particular, we study this task for a range of hand designs with links and joints removed as well as link length extensions. With only a single set of network weights, GET-Zero is able to control a wide range of hand designs and generalize to new designs not seen during training. The main idea is that by conditioning our model on the hand design, we can create a policy that is aware of how each finger is structured and can act accordingly. Training Embodiments

For the original hand design with all joints and links present (seen during training), GET-Zero achieves higher average rotational velocity relative to the baseline. The baseline has the same overall model architecture as GET, except without the graph encoding nor self-modeling.

In simulation, we train on a set of 44 embodiments with different graph variations consisting of joints and links removed. We observe that GET-Zero achieves a 10% performance improvement over the baseline on these training embodiments, which enables our method to match the rotation performance (99%) of 44 embodiment-specific RL policies despite using a single set of network weights to control every design. GET-Zero is capable of zero-shot control of hand designs not seen during training. In this section, we show performance on embodiments that have combinations of removed joints and links that were not seen during training (new graph variations). Unseen Graph and Link Length Extensions (Zero-shot)

Across 10 embodiments with these variations, we find that GET-Zero achieves a 20% performance improvement over the best performing baseline. Failure Cases

The sim-to-real zero-shot control of a previously unseen embodiment is challenging for contact-rich tasks, especially with both graph and link length variations present. We observe a few failure cases with GET-Zero.

The hand below (unseen graph and link length extensions) has a missing index finger that creates a gap where the cube can fall through. This compounds any control errors which can come from the fact that link length extensions were never seen during training.

Falling cube (GET-Zero + unseen graph and link length extensions)


Next, the hand below (also with unseen graph and link length extensions) has difficulty rotating the cube. This is partially due to a pinky finger that only has a single joint, making it difficult to start the rotation cycle. Additionally, GET-Zero uses a proprioception-only input (no visual input nor explicit cube state provided) to minimize the sim-to-real gap, but this makes it challenging for the policy to adapt to repeated failures.
